# Agent 核心能力模型

> 能力模型是理解和设计 Agent 系统的基础框架。它将 Agent 所需的各项能力分层组织，帮助工程师在系统设计阶段就明确能力边界、识别短板、选择合理的架构方案。本文从学术与工业界的多种模型出发，提出一个面向工程实践的五层能力模型，并讨论评估方法与设计启示。

## 1. 能力模型概述

### 为什么需要一个能力模型

在构建 Agent 系统时，工程师面临的第一个核心问题不是"用什么框架"，而是"我需要这个 Agent 具备哪些能力"。一个清晰的能力模型能帮助团队在项目早期就回答三个关键问题：当前 LLM 的哪些能力可以直接利用、哪些能力需要通过工具和架构来补偿、以及哪些能力在当前技术条件下根本无法实现。没有能力模型作为指导，工程师往往会陷入两个极端——要么高估 LLM 的能力导致系统不可靠，要么低估其潜力而过度依赖硬编码逻辑。能力模型还为系统评估提供了结构化的维度，让团队能够有针对性地识别瓶颈并制定改进策略。

### 不同来源的能力模型对比

学术界最具影响力的能力模型来自 Wang et al. 的综述论文 [Wang et al., 2024] A Survey on Large Language Model based Autonomous Agents. Frontiers of Computer Science. 该框架将 Agent 的构成分解为四个模块：Profile（角色定义，决定 Agent 的身份与行为边界）、Memory（记忆机制，包括短期工作记忆与长期经验存储）、Planning（规划能力，涵盖任务分解、策略选择与反思调整）、Action（行动执行，包括工具调用与环境交互）。这一框架的优势在于其完备性和学术严谨性，为后续大量研究提供了统一的讨论语言。

工业界的视角则更关注工程可落地性。Anthropic 在 2024 年 12 月发布的 [Anthropic, 2024] Building Effective Agents. anthropic.com/research/building-effective-agents 中提出了 Augmented LLM 的概念——Agent 系统的基础构建块是一个通过检索（Retrieval）、工具（Tools）和记忆（Memory）增强的 LLM。Anthropic 特别强调，最成功的 Agent 实现都使用了简单、可组合的模式，而非复杂的框架。他们定义了从 Prompt Chaining、Routing、Parallelization 到 Orchestrator-Workers、Evaluator-Optimizer 的五种工作流模式，以及真正自主循环的 Agent 模式。

OpenAI 在 2025 年初发布的 [OpenAI, 2025] A Practical Guide to Building Agents 中进一步简化为三个核心要素：Model（选择具备足够推理能力的模型）、Tools（定义 Agent 可以执行的操作集合）、Instructions（通过系统提示词定义行为规范）。OpenAI 的框架更加面向产品团队，强调从单 Agent 起步、逐步增加复杂度的渐进式路径。

Andrew Ng 在 2024 年提出的 Agentic Design Patterns 则从设计模式角度切入 [Andrew Ng, 2024] What's Next for AI Agentic Workflows. Sequoia Capital AI Ascent / Snowflake BUILD 2024 Keynote，识别出四种基础模式：Reflection（自我反思与改进）、Tool Use（外部工具调用）、Planning（任务规划与分解）、Multi-Agent Collaboration（多 Agent 协作）。这些模式彼此正交、可自由组合，为工程师提供了一套实用的设计词汇表。

本文综合以上框架，提出一个面向工程实践的五层能力模型：基础能力层（LLM 本身）、扩展能力层（工具与环境）、认知能力层（规划与元认知）、协作能力层（社会智能）、学习能力层（适应与进化）。这种分层结构既反映了能力的依赖关系（上层依赖下层），也对应了工程实现的复杂度递增。

## 2. 基础能力层：LLM 作为大脑

基础能力层是整个 Agent 系统的根基，对应 LLM 本身所具备的核心智能。这些能力不依赖外部工具或架构设计，而是模型内化的认知功能。理解这一层的能力边界，是决定后续所有架构决策的前提。

### 语言理解与生成

语言理解与生成是 LLM 最基本也是最强大的能力。在理解端，模型需要处理自然语言的多义性、隐含信息、语境依赖和修辞手法，从用户的模糊表述中提取精确意图。在生成端，模型需要产出语法正确、语义连贯、风格适当的文本，覆盖从代码到散文、从技术文档到日常对话的广泛场景。对于 Agent 来说，语言能力不仅是"能说会写"，更是一种思维工具——Agent 的内部推理、计划制定、工具参数生成都以语言为载体。语言生成的质量直接决定了 Agent 与人类协作的效率和用户体验。

### 推理能力

推理是将 LLM 从"文本补全器"提升为"智能决策者"的关键能力。LLM 展现出的推理能力主要包含三种类型：演绎推理（从一般规则推导出具体结论，如根据 API 文档推断正确的调用方式）、归纳推理（从多个具体实例中总结出一般规律，如从错误日志中识别模式）、以及类比推理（将一个领域的知识迁移到另一个领域，如将设计模式从一种编程语言迁移到另一种）。Chain-of-Thought（CoT）提示技术 [Wei et al., 2022] 的发现表明，通过引导模型显式展开推理步骤，可以显著提升在数学、逻辑和常识推理任务上的表现。近期的 o1、DeepSeek-R1 等推理模型则通过强化学习进一步增强了模型的长链推理能力，使其在复杂问题上的表现更加稳健。

### 指令遵循（Instruction Following）

指令遵循能力决定了 Agent 的可控性和可靠性。一个优秀的 Agent 需要精确理解并执行复杂的多层指令：系统级指令（定义 Agent 的角色和行为边界）、任务级指令（描述当前需要完成的目标）、以及格式化指令（规定输出的结构和约束）。通过 [RLHF](../../appendix/glossary.md#rlhf)（Reinforcement Learning from Human Feedback）和指令微调（Instruction Tuning），现代 LLM 在指令遵循方面取得了巨大进步。然而，当指令之间存在冲突、指令过于复杂或模糊时，模型仍然可能产生偏离，这是 Agent 工程中需要通过 Guardrails 和验证机制来额外保障的环节。

### 上下文学习（In-Context Learning）

上下文学习是 LLM 最令人惊叹的涌现能力之一 [Wei et al., 2022] Emergent Abilities of Large Language Models. arXiv:2206.07682。它使得模型无需更新权重，仅通过在提示中提供少量示例（Few-shot）或任务描述（Zero-shot），就能适应全新的任务。对于 Agent 来说，上下文学习意味着系统可以在运行时动态调整行为——通过在上下文中注入相关的历史交互记录、领域知识片段或行为示例，Agent 能够快速适应特定用户、特定领域或特定工作流。上下文窗口的大小直接制约了这一能力的上限：窗口越大，Agent 能够利用的即时信息就越丰富，但也带来了注意力稀释和成本增加的挑战。

### 多模态感知

随着 GPT-4V、Claude 3、Gemini 等多模态模型的成熟，Agent 的感知能力已从纯文本扩展到视觉（图像理解、截图分析、文档 OCR）、音频（语音识别、音频理解）等多种模态。多模态感知极大拓展了 Agent 的应用场景：GUI Agent 可以"看到"屏幕内容并操作界面，代码 Agent 可以理解架构图和 UML 图，数据分析 Agent 可以直接解读图表。从工程角度看，多模态输入增加了系统的复杂性——需要处理不同模态的对齐、不同模态信息的优先级冲突，以及多模态输入带来的延迟和成本。

### 能力如何随规模与训练方法提升

研究表明，LLM 的许多核心能力并非随模型规模线性提升，而是呈现出"涌现"（Emergence）现象——当模型参数超过某个阈值时，特定能力会突然出现。例如，Chain-of-Thought 推理、代码生成、多语言翻译等能力都在约 100B 参数规模时显著涌现。除了规模之外，训练方法同样关键：指令微调（Instruction Tuning）显著提升了零样本任务完成能力，[RLHF](../../appendix/glossary.md#rlhf) 改善了输出质量和安全性，而代码数据的加入则被认为增强了模型的逻辑推理能力。对于 Agent 工程师来说，理解这些规律有助于选择合适的基座模型：并非所有任务都需要最大的模型，关键是匹配任务所需的能力层级。

## 3. 扩展能力层：工具与环境交互

扩展能力层解决的核心问题是：LLM 本身是一个"大脑"，但大脑需要"手脚"才能作用于真实世界。通过工具调用和环境交互，Agent 的能力从"思考和表达"扩展到"感知和行动"，实现了从语言模型到行动者的跨越。

### 工具调用（Function Calling）

工具调用是 Agent 最基础的扩展机制，它让 LLM 能够调用外部函数来获取实时信息、执行计算或触发操作。现代 LLM 的 Function Calling 能力已经相当成熟，核心流程包含三步：模型理解工具的描述（通常以 JSON Schema 形式提供）、根据当前任务决定是否调用以及如何构造参数、最后解析工具返回的结果并融入后续推理。工具调用的工程挑战在于：如何设计清晰、无歧义的工具接口描述，使模型能够准确选择和使用工具；如何处理工具调用失败、超时或返回异常数据的情况；以及当可用工具数量很大时，如何帮助模型在众多选项中做出正确选择。Berkeley Function Calling Leaderboard（BFCL）[Gorilla Project, 2024] 提供了系统化的评估框架，涵盖简单调用、并行调用、多步调用和 Agentic 场景下的工具使用能力。

### 代码执行

代码执行赋予 Agent 精确计算和动态创造的能力。与直接输出文本答案不同，Agent 可以生成代码、在沙箱环境中执行、观察输出结果，然后基于结果调整策略。这种"生成-执行-观察"的循环模式特别适合数据分析、数学计算、图表生成、文件处理等需要精确性的任务。代码执行的关键工程考量包括：安全隔离（沙箱机制防止恶意或错误代码影响宿主系统）、资源限制（控制 CPU、内存、执行时间防止资源耗尽）、状态持久化（跨多轮交互保持执行上下文）、以及错误恢复（当代码执行失败时，Agent 能从错误信息中学习并修正代码）。

### 信息检索

信息检索扩展了 Agent 的知识边界，使其能够访问训练数据之外的最新信息和专有知识。最典型的模式是 [RAG](../../appendix/glossary.md#rag)（Retrieval-Augmented Generation），它将检索到的相关文档片段注入到模型的上下文中，作为回答问题的依据。在 Agent 系统中，信息检索不仅是被动的"查资料"，更是主动的信息获取策略——Agent 会根据当前任务需要，自主决定何时检索、检索什么、从哪里检索。工程上需要解决的问题包括：检索的准确性（如何确保召回的信息确实相关）、检索的时效性（如何获取最新信息）、多源融合（如何整合来自搜索引擎、知识库、数据库等不同来源的信息），以及检索结果的可信度判断（Agent 需要识别信息的来源可靠性）。

### 环境操作

环境操作是 Agent 从"咨询助手"进化为"执行者"的关键能力。根据操作对象的不同，可以分为几个层次：文件系统操作（读写文件、管理目录结构，是代码 Agent 的基础能力）、API 交互（调用 REST/GraphQL 等 Web API 完成业务操作）、浏览器操作（导航网页、填写表单、点击按钮，实现 Web 自动化）、以及 GUI 操作（操作桌面应用程序的图形界面）。环境操作的核心挑战是状态管理和错误处理：真实环境是有状态的（文件一旦删除就无法恢复，API 调用可能产生不可逆的副作用），Agent 需要理解操作的后果并具备适当的谨慎性。Anthropic 在其指南中特别强调了 Human-in-the-loop 机制的重要性——对于高风险操作，让人类确认后再执行。

## 4. 认知能力层：规划与元认知

认知能力层是将 Agent 从"工具的被动使用者"提升为"任务的主动管理者"的关键。如果说基础层和扩展层提供了"能做什么"，认知层则决定了"如何做"和"做得好不好"。这一层对应人类认知中的执行功能（Executive Function），是 Agent 系统复杂度和可靠性的分水岭。

### 任务分解（Decomposition）

面对复杂的用户目标，Agent 需要将其拆解为可执行的子任务序列。有效的任务分解要求 Agent 理解任务的层次结构（哪些是顶层目标、哪些是中间步骤、哪些是原子操作）、子任务之间的依赖关系（哪些可以并行、哪些必须串行），以及每个子任务所需的能力和资源。ReAct 框架 [Yao et al., 2023] ReAct: Synergizing Reasoning and Acting in Language Models. ICLR 2023 开创性地展示了如何通过交错生成推理轨迹（Thought）和行动（Action）来实现隐式的任务分解。在实际工程中，任务分解的粒度选择是一个关键决策：粒度太粗会导致单步执行过于复杂而容易失败，粒度太细则会增加规划开销和步骤间的信息传递成本。

### 策略选择与规划（Planning）

规划能力使 Agent 能够在执行前"先想后做"，根据目标和约束生成行动序列。LLM Agent 的规划可以分为前向规划（从当前状态出发，逐步推进到目标状态）和后向规划（从目标状态回溯，确定需要满足的前提条件）。在复杂场景中，Agent 可能需要在多个可行方案中做选择——这涉及对每种方案的成本、风险、成功概率的估计。Tree-of-Thought [Yao et al., 2023] 等方法通过让模型探索多条推理路径并进行比较，提升了规划的质量。工程实践中，常见的策略是将规划与执行分离：先生成一个高层计划，然后逐步执行，在每一步根据实际结果动态调整后续计划。

### 自我监控（Self-monitoring）

自我监控是 Agent 在执行过程中持续检查"我是否在正确的轨道上"的能力。这包括几个方面：进度追踪（当前完成了哪些子任务、距离目标还有多远）、偏差检测（执行结果是否符合预期、是否偏离了原始目标）、以及资源监控（是否超出了 token 预算、时间限制或 API 调用配额）。缺乏自我监控能力的 Agent 容易陷入"执行惯性"——即使已经偏离正轨也继续盲目执行，甚至在循环中反复尝试已经证明无效的策略。工程上可以通过显式的检查点（Checkpoint）机制、执行日志的自动审查、以及周期性的目标重确认来增强这一能力。

### 反思与错误修正（Reflection & Self-correction）

反思是 Agent 从失败中学习的核心机制。Reflexion 框架 [Shinn et al., 2023] Reflexion: Language Agents with Verbal Reinforcement Learning. NeurIPS 2023 证明了一种无需更新模型权重的学习方式：Agent 在执行失败后，生成自然语言形式的反思总结（"我上次失败是因为……下次应该……"），并将这些反思保存在情景记忆中，用于指导后续尝试。这种"语言强化学习"使得 Agent 能够在不重新训练的情况下逐步改进行为。在工程实现中，反思机制需要解决几个问题：如何准确归因失败原因（而不是归因于无关因素）、如何避免过度修正（矫枉过正）、以及如何将反思经验泛化到相似但不同的场景中。

### 目标管理与优先级判断

在真实场景中，Agent 往往面临多个目标同时存在且相互竞争的情况。目标管理能力要求 Agent 能够维护一个目标栈或目标树，理解目标之间的优先级关系、时间约束和资源冲突，并在必要时做出取舍。例如，一个开发 Agent 在编写代码时需要同时平衡功能正确性、代码质量、性能要求和交付时间等多个目标。当某个子目标被证明无法实现时，Agent 需要能够优雅地放弃并调整整体策略，而不是陷入无限重试。这种能力在当前的 Agent 系统中仍然是一个相对薄弱的环节，往往需要通过 Prompt 工程和架构设计来补偿。

## 5. 协作能力层：社会智能

协作能力层关注的是 Agent 在社会性情境中的行为——无论是与人类协作还是与其他 Agent 协同。随着 Multi-Agent 系统的兴起，这一层能力的重要性日益凸显。

### 角色理解与扮演

在多 Agent 系统中，每个 Agent 通常承担一个特定的角色（如产品经理、开发工程师、测试工程师、代码审查者等）。角色理解能力要求 Agent 不仅知道自己的职责范围，还能准确把握角色之间的互动模式和信息流动方式。一个好的角色扮演不是简单的"你是一个 X"的声明，而是需要模型内化角色的知识背景、思维方式、关注重点和沟通风格。在人机协作场景中，Agent 还需要理解人类用户的角色——判断对方是技术专家还是非技术人员、是决策者还是执行者——并相应调整自己的表达方式和信息粒度。

### 信息共享与通信

在多 Agent 系统中，Agent 之间需要高效地交换信息来协调行动。通信能力包括：信息的结构化表达（将自己的工作成果、发现或判断以其他 Agent 能理解的方式表述）、信息的选择性共享（判断哪些信息对协作伙伴是必要的、避免信息过载）、以及通信协议的遵循（在什么时机、通过什么渠道、以什么格式传递信息）。工程上，这涉及到消息格式的标准化、共享状态的管理、以及异步通信机制的设计。Anthropic 的 Orchestrator-Workers 模式就是一种典型的通信架构：编排者 Agent 负责分发任务并收集结果，工作者 Agent 专注于执行并汇报。

### 冲突协商

当多个 Agent 的判断或方案产生分歧时，系统需要一种机制来解决冲突。冲突可能源于信息不对称（各 Agent 看到不同的数据）、目标差异（各角色的优化目标不同）、或能力限制（某个 Agent 的方案在另一个 Agent 的领域内不可行）。冲突解决策略包括：投票机制（多数决）、仲裁机制（由上层 Agent 或人类裁决）、辩论机制（各方陈述理由后达成共识）、以及妥协机制（寻找各方都能接受的折中方案）。在实践中，MIT 的 Society of Mind 和 ChatDev 等项目展示了通过结构化的多 Agent 对话来协商和解决分歧的可能性。

### 任务委派与接受

任务委派是层次化 Agent 系统中的核心交互模式。委派方需要具备的能力包括：准确评估子任务的复杂度和所需能力、选择合适的执行者、提供充分但不冗余的上下文信息、以及设定清晰的完成标准。接受方则需要：准确理解任务要求、评估自己是否有能力完成、在遇到阻碍时及时求助或汇报、以及在完成后提供结构化的结果摘要。工程上，这对应了 Agent 框架中的子 Agent 调度（如 OpenAI 的 Handoff 机制）——主 Agent 将特定子任务"移交"给专门的子 Agent 处理，子 Agent 完成后将结果返回主 Agent。

## 6. 学习能力层：适应与进化

学习能力层关注 Agent 如何随着时间和经验变得更好。与传统 ML 系统通过梯度更新学习不同，LLM Agent 的学习主要通过上下文、记忆和外部反馈机制实现，这种"无权重更新的学习"是 Agent 系统的独特特征。

### 上下文内学习（单次会话内）

上下文内学习是 Agent 在单次会话过程中逐步积累信息、调整行为的能力。随着对话的进行，Agent 通过已有的交互历史了解用户的偏好、任务的具体约束和环境的特点，从而在后续交互中做出更准确的判断。例如，一个编码 Agent 在最初几轮交互中可能使用通用的编码风格，但随着观察到用户的代码仓库风格，它会自适应地调整。上下文内学习的效率取决于模型的上下文窗口大小和注意力机制的有效性——信息在窗口中的位置、密度和组织方式都会影响学习效果。

### 跨会话记忆与知识积累

跨会话记忆解决的是"Agent 不应该每次重新认识用户"的问题。通过持久化的记忆系统，Agent 能够记住用户的长期偏好、历史决策、项目背景等信息，并在新会话中利用这些知识。记忆系统的设计涉及几个关键问题：记什么（如何从大量交互中提取值得记忆的信息）、怎么存（向量数据库、结构化存储还是自然语言摘要）、何时用（如何在合适的时机召回相关记忆而不干扰当前任务），以及如何遗忘（过时或错误的记忆如何被更新或删除）。一个好的记忆系统应该像人类记忆一样具有选择性和层次性——重要的核心信息高度可用，琐碎的细节按需检索。

### 策略优化（从反馈中改进）

策略优化是 Agent 从每次任务执行的结果中学习"什么方法更有效"的能力。这里的反馈可以来自多种渠道：任务执行的客观结果（代码是否通过测试、部署是否成功）、人类的显式反馈（用户表示满意或不满意）、以及自我评估（Agent 反思自己的方案是否有更好的替代）。Reflexion 框架展示了一种纯语言化的策略优化方式，而更系统化的方法可能包括：维护一个"策略库"记录不同类型任务的最佳实践、构建"失败案例库"避免重复犯错、以及通过 A/B 比较不同策略的效果来渐进式优化。在工程实现中，策略优化通常需要一个评估机制（知道什么是"好"的结果）和一个归因机制（知道是哪个决策导致了好或坏的结果）。

### 人类偏好对齐

人类偏好对齐是 Agent 持续适应特定用户或组织工作方式的能力。不同用户对"好的帮助"有不同的定义：有人偏好详细解释，有人只要结果；有人希望 Agent 主动提建议，有人希望它严格执行指令；有人在乎代码的优雅性，有人更关心快速交付。偏好对齐可以在多个层面实现：显式偏好（用户直接说明"我喜欢 X 风格"）、隐式偏好（Agent 从用户的修改和反馈中推断偏好）、以及组织级偏好（团队的编码规范、流程标准等）。一个成熟的 Agent 系统能够建立用户模型（user model），随着交互的增加不断更新对用户偏好的理解，从而提供越来越个性化的服务。这与推荐系统中的用户画像有异曲同工之处，但在 Agent 场景下需要更精细的粒度和更强的可解释性。

## 7. 能力评估框架

### 如何度量每一层的能力水平

评估 Agent 的能力需要多维度、多层次的框架，不能用单一指标衡量。对于基础能力层，可以使用标准 NLP benchmark（如 MMLU、HumanEval、MATH）评估语言理解、代码生成和推理能力。对于扩展能力层，Berkeley Function Calling Leaderboard（BFCL）等专门的工具调用评测集提供了标准化的评估方法，包括参数生成准确率、工具选择正确率、多步工具链执行成功率等指标。对于认知能力层，可以通过复杂任务完成率、规划步骤的合理性、错误恢复能力等指标来评估，AgentBench [Zheng et al., 2024] 等综合性 benchmark 提供了跨环境的评估框架。协作能力层和学习能力层的评估相对更困难，通常需要设计特定的交互场景和纵向实验来衡量。一个实用的做法是为每个能力维度定义清晰的等级标准（如 L1-L5），并建立对应的测试用例集。

### 能力瓶颈识别

系统整体能力遵循"木桶效应"——取决于最弱的环节。一个 Agent 可能拥有极强的推理能力，但如果工具调用不稳定，最终的任务完成率依然会很低。识别能力瓶颈的方法包括：错误归因分析（将失败案例归类到不同能力层）、消融实验（逐步移除某个能力模块观察影响）、以及端到端性能与各环节性能的对比分析。在实际工程中，最常见的瓶颈往往不是模型本身的能力不足，而是能力层之间的衔接问题——比如模型有能力做正确的规划，但在将规划转化为具体的工具调用序列时出现偏差。识别这些"接口瓶颈"需要细粒度的可观测性（observability）设计。

### 能力组合效应

某些能力的组合会产生超线性的效果，这在系统设计中是重要的杠杆点。例如，推理能力 + 工具调用的组合使得 Agent 不仅能执行预定义的工具序列，还能根据中间结果动态调整后续步骤，这就是 ReAct 模式的核心洞察。类似地，反思能力 + 记忆系统的组合使得 Agent 能够从过去的错误中真正"学到"东西，而不仅仅是在单次交互中修正错误。规划能力 + 多 Agent 协作的组合使得复杂任务可以被有效分解并并行执行。理解这些组合效应有助于在有限资源下做出最优的系统设计决策——优先投资那些能产生乘数效应的能力组合。

## 8. 工程启示

### 设计 Agent 时应先评估需要哪些能力层

在开始构建 Agent 系统之前，工程师应该首先对目标任务进行能力需求分析。不是所有任务都需要所有能力层：一个简单的问答 Agent 可能只需要基础能力层；一个代码助手需要基础层 + 扩展层（工具调用）+ 部分认知层（规划）；一个复杂的研发流程自动化系统则可能需要全部五层能力。过度设计（over-engineering）和能力不足（under-engineering）都是常见的陷阱。通过能力模型的框架进行前期评估，可以避免在不必要的能力上投入过多资源，也可以提前识别哪些能力是项目的关键风险点。实践中，建议从最简单的架构开始，当遇到能力瓶颈时再有针对性地增加复杂度。

### 能力不足的补偿策略

当某一层的能力存在不足时，可以通过其他机制来补偿。工具补偿认知不足是最常见的策略：如果模型的数学推理能力不够强，可以提供计算器工具；如果模型的知识过时，可以通过搜索工具接入实时信息。多 Agent 补偿单 Agent 的局限也是有效的方法：将一个复杂任务分配给多个专注于不同子任务的 Agent，每个 Agent 只需要擅长其特定领域。人类介入补偿自主性不足适用于高风险场景：在 Agent 不确定的决策点引入人工审批，既保证了安全性，又充分利用了 Agent 在确定性环节的效率优势。此外，流程约束补偿规划能力不足也很实用：通过预定义的 workflow 模板来限制 Agent 的行动空间，降低对自主规划能力的要求。

### 能力提升路径

提升 Agent 系统能力有一条渐进式的路径：Prompt 优化 - 工具增强 - Fine-tuning - 架构升级。Prompt 优化是成本最低、见效最快的手段，通过改进系统提示、添加 few-shot 示例、优化工具描述等方式，通常能在不改变其他组件的情况下带来显著的性能提升。工具增强是下一步，通过为 Agent 提供更好的工具（更精准的 API、更丰富的信息源、更强大的执行环境）来扩展其能力边界。Fine-tuning 适用于 Prompt 优化已经达到瓶颈的场景，通过在特定领域数据上微调模型来提升基础能力。架构升级则是最重大的变更，包括引入多 Agent 协作、添加记忆系统、实现自适应规划等。这条路径从简单到复杂、从低成本到高成本，建议工程师按顺序尝试，每一步都充分验证效果后再决定是否进入下一步。
